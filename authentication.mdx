## Overview

HolySuck API uses multiple authentication methods to ensure secure access to your resources. Choose the method that best fits your use case.

## API Key Authentication

The simplest way to authenticate with HolySuck API is using API keys.

### Getting Your API Key

1. Log into your [HolySuck Dashboard](https://app.holysuck.com)
2. Navigate to **Settings** â†’ **API Keys**
3. Click **Create New API Key**
4. Give your key a descriptive name
5. Copy the key immediately (it won't be shown again)

### Using API Keys

Include your API key in the `Authorization` header with the `Bearer` prefix:

<CodeGroup>

```bash cURL
curl -X GET "https://api.holysuck.com/v1/users/me" \
  -H "Authorization: Bearer your_api_key_here"
```

```javascript JavaScript
const headers = {
  'Authorization': 'Bearer your_api_key_here',
  'Content-Type': 'application/json'
};
```

```python Python
headers = {
    'Authorization': 'Bearer your_api_key_here',
    'Content-Type': 'application/json'
}
```

```go Go
req.Header.Add("Authorization", "Bearer your_api_key_here")
```

</CodeGroup>

<Warning>
**Keep your API keys secure!** Never commit them to version control or expose them in client-side code.
</Warning>

## OAuth 2.0

For applications that need to act on behalf of users, use OAuth 2.0 flow.

### Authorization Code Flow

1. **Redirect to Authorization URL**
   ```
   https://api.holysuck.com/oauth/authorize?
     client_id=your_client_id&
     redirect_uri=your_redirect_uri&
     response_type=code&
     scope=read write&
     state=random_state_string
   ```

2. **Exchange Code for Token**
   ```bash
   curl -X POST "https://api.holysuck.com/oauth/token" \
     -H "Content-Type: application/x-www-form-urlencoded" \
     -d "grant_type=authorization_code" \
     -d "client_id=your_client_id" \
     -d "client_secret=your_client_secret" \
     -d "code=authorization_code" \
     -d "redirect_uri=your_redirect_uri"
   ```

3. **Use Access Token**
   ```bash
   curl -X GET "https://api.holysuck.com/v1/users/me" \
     -H "Authorization: Bearer access_token"
   ```

### Token Response

```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "scope": "read write"
}
```

## JWT Tokens

For server-to-server communication, you can use JWT tokens with your service account.

### Creating JWT

```javascript
const jwt = require('jsonwebtoken');

const payload = {
  iss: 'your_service_account_id',
  sub: 'your_service_account_id',
  aud: 'https://api.holysuck.com',
  exp: Math.floor(Date.now() / 1000) + 3600, // 1 hour
  iat: Math.floor(Date.now() / 1000)
};

const token = jwt.sign(payload, 'your_private_key', { algorithm: 'RS256' });
```

### Using JWT

```bash
curl -X GET "https://api.holysuck.com/v1/users/me" \
  -H "Authorization: Bearer your_jwt_token"
```

## Rate Limiting

All authentication methods are subject to rate limits:

- **API Key**: 1000 requests per minute
- **OAuth Token**: 5000 requests per minute  
- **JWT Token**: 10000 requests per minute

Rate limit headers are included in all responses:

```http
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 999
X-RateLimit-Reset: 1609459200
```

## Scopes

OAuth and JWT tokens support different scopes:

| Scope | Description |
|-------|-------------|
| `read` | Read access to user data |
| `write` | Write access to user data |
| `admin` | Administrative access |
| `analytics` | Access to analytics data |
| `webhooks` | Manage webhooks |

## Best Practices

<Accordion title="Environment Variables">
Store your API keys and secrets in environment variables:

```bash
export HOLYSUCK_API_KEY="your_api_key"
export HOLYSUCK_CLIENT_SECRET="your_client_secret"
```
</Accordion>

<Accordion title="Token Rotation">
Regularly rotate your API keys and update your applications:

1. Create a new API key
2. Update your applications
3. Delete the old key
</Accordion>

<Accordion title="Monitoring">
Monitor your API usage in the dashboard:

- Track failed authentication attempts
- Monitor rate limit usage
- Review access patterns
</Accordion>

## Error Handling

Common authentication errors:

| Status Code | Error | Description |
|-------------|-------|-------------|
| 401 | `invalid_token` | Token is invalid or expired |
| 401 | `missing_token` | No authorization header provided |
| 403 | `insufficient_scope` | Token lacks required permissions |
| 429 | `rate_limit_exceeded` | Too many requests |

Example error response:

```json
{
  "error": "invalid_token",
  "error_description": "The access token provided is expired, revoked, malformed, or invalid.",
  "status": 401
}
```
